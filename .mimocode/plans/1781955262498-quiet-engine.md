# Shared Folder — Complete Architectural Review

## 1. Why Does the Project Need a `shared` Folder?

This is a **monorepo** with three packages: `frontend/` (Next.js), `backend/` (Express + Supabase), and `shared/`. The `shared/` folder exists to provide a **single source of truth** for code that both frontend and backend need — Zod schemas, TypeScript types, error classes, constants, and utility functions — so both sides agree on data shapes without duplicating definitions.

**In practice, the shared folder is currently only used by the backend. The frontend has zero imports from shared.**

---

## 2. Which Files Are Used by Both Frontend and Backend?

**None.** Only the backend imports from shared. The frontend has the tsconfig path alias configured but never uses it.

| Shared Export | Backend Uses It? | Frontend Uses It? |
|---|---|---|
| `AppError`, `UnauthorizedError`, `StorageError`, `AIError`, `MissingConfigError` | Yes (10 files) | **No** |
| `ScanResult`, `Profile`, `Appointment` types | Yes (5 files) | **No** |
| `SUPABASE_BUCKETS`, `ANTHROPIC_MODELS` | Yes (2 files) | **No** |
| `LOCALITIES`, `SERVICES`, `BOOKING_SERVICES`, `TIME_SLOTS`, `BOOKED_SLOTS` | **No** | **No** (duplicated locally) |
| `Salon`, `Stylist`, `Testimonial`, `BookingService`, etc. | **No** | **No** (types exist but aren't imported from shared) |
| `cn`, `formatDate`, `formatCurrency`, `generateTimeSlots` | **No** | **No** (frontend has its own `cn`) |
| All Zod schemas (`uploadScanImageInput`, etc.) | **No** (types used, schemas not validated) | **No** |

## 3. Per-File Analysis

### `shared/schemas/scan.schema.ts`

- **What it does:** Defines Zod validation schemas for the GlamAI scan feature (upload, analyze, get scans) + 5 error classes (`AppError`, `UnauthorizedError`, `MissingConfigError`, `StorageError`, `AIError`)
- **Imported by:** 10 backend files (error.middleware, glamai.routes, scan.service, analysis.service, storage.ts, scan.repository, user.repository, appointment.repository, user.service, appointment.service, anthropic/client)
- **Actually used:** The **error classes** and `ScanResult` type are heavily used. The **Zod schemas themselves** (`uploadScanImageInput`, `analyzeScanInput`, etc.) are **never invoked** — the `validate` middleware exists but is not wired into any route.
- **Can it be removed?** No — the error classes are critical infrastructure. But the Zod schemas could be moved or the validation middleware wired in.

### `shared/types/index.ts`

- **What it does:** Re-exports Zod-inferred types from scan.schema.ts, plus defines 9 frontend-oriented interfaces (`Salon`, `Stylist`, `Testimonial`, `BookingService`, `BookingStylist`, `TourSalon`, `ExclusiveService`, `Profile`, `Appointment`)
- **Imported by:** Backend uses `Profile` (2 files), `Appointment` (2 files), `ScanResult` (via schemas, 2 files). Frontend imports **none**.
- **Actually used:** `Profile`, `Appointment`, `ScanResult` are used by backend. The other 6 interfaces (`Salon`, `Stylist`, `Testimonial`, `BookingService`, `BookingStylist`, `TourSalon`, `ExclusiveService`) are **dead code**.
- **Can it be removed?** No — `Profile`, `Appointment`, `ScanResult` are needed. The 6 unused interfaces can be removed.

### `shared/constants/index.ts`

- **What it does:** Exports 8 constants — locality lists, service lists, booking services, time slots, booked slots, API version, Supabase bucket name, Anthropic model ID
- **Imported by:** Backend uses `SUPABASE_BUCKETS` (1 file) and `ANTHROPIC_MODELS` (1 file). Frontend imports **none**.
- **Actually used:** 2 of 8 constants are used. `LOCALITIES`, `SERVICES`, `BOOKING_SERVICES`, `TIME_SLOTS`, `BOOKED_SLOTS`, `API_VERSION` are **dead code** in shared (frontend duplicates `LOCALITIES`, `SERVICES`, `BOOKING_SERVICES`, `TIME_SLOTS`, `BOOKED_SLOTS` locally in `src/data/`).
- **Can it be removed?** Partially. `SUPABASE_BUCKETS` and `ANTHROPIC_MODELS` are required. The rest are unused from shared.

### `shared/utils/index.ts`

- **What it does:** Exports `cn()` (Tailwind class merger), `formatDate()`, `formatCurrency()`, `generateTimeSlots()`
- **Imported by:** **Nobody.** Frontend has its own identical `cn()` in `src/lib/utils.ts`.
- **Actually used:** Zero imports. Completely dead code.
- **Can it be removed?** Yes — entirely.

### `shared/index.ts`

- **What it does:** Barrel re-export of all sub-modules
- **Imported by:** Nobody (backend imports directly from sub-modules via relative paths)
- **Can it be removed?** Yes — it's unused.

## 5. What Belongs in Shared?

| Category | Belongs in Shared? | Reason |
|---|---|---|
| **TypeScript types** (domain models like `Profile`, `Appointment`) | ✅ Yes | Both sides need to agree on data shapes |
| **Zod schemas** (API contract validation) | ✅ Yes | Shared validation ensures frontend sends what backend expects |
| **DTOs** (Data Transfer Objects) | ✅ Yes | If you have request/response shapes |
| **Constants** (API URLs, bucket names, model IDs) | ✅ Yes | Avoids magic strings in both codebases |
| **Enums** (status values, roles) | ✅ Yes | Single source of truth for string unions |
| **Utility functions** | ⚠️ Only if shared | `cn()` is frontend-only (Tailwind). `formatDate()` could be shared if both need it. Don't put frontend-only utils in shared. |
| **Validation logic** | ✅ Yes | This is the whole point of shared Zod schemas |
| **Error classes** | ✅ Yes | Backend uses them; frontend should too for API error handling |

## 8. Benefits of Shared vs Duplicating

### With Shared (current intent):
- **Single source of truth** — change `Profile` type once, both sides get it
- **Zod validation** — frontend schemas automatically validate what backend expects
- **Error consistency** — same error hierarchy everywhere
- **Type safety** — TypeScript catches mismatches at compile time

### Duplicating (current reality):
- Frontend **already duplicates** constants (`LOCALITIES`, `SERVICES`, etc.)
- Frontend **already duplicates** `cn()` utility
- **Risk:** If shared constants change, frontend drifts out of sync
- **Waste:** ~60% of shared code is dead weight

### Verdict: Shared is the right architecture, but it's **under-utilized**. The frontend should migrate to import from shared instead of duplicating.

## 10. Recommended Structure

```
shared/
├── index.ts                    # Barrel export (update to match new structure)
├── package.json
├── tsconfig.json
├── schemas/
│   ├── index.ts                # Re-export all schemas
│   ├── scan.schema.ts          # ✅ KEEP — error classes + scan types
│   ├── booking.schema.ts       # 🆕 ADD — booking validation (when backend implements)
│   └── user.schema.ts          # 🆕 ADD — user profile validation
├── types/
│   ├── index.ts                # Re-export all types
│   ├── salon.ts                # 🆕 ADD — Salon, Stylist, etc. (for when backend serves salon data)
│   ├── user.ts                 # ✅ KEEP — Profile, Appointment
│   └── scan.ts                 # ✅ KEEP — re-export from schemas
├── constants/
│   ├── index.ts                # Re-export all
│   ├── locations.ts            # ✅ KEEP — LOCALITIES (frontend should import from here)
│   ├── services.ts             # ✅ KEEP — SERVICES, BOOKING_SERVICES (frontend should import from here)
│   ├── booking.ts              # ✅ KEEP — TIME_SLOTS, BOOKED_SLOTS
│   ├── api.ts                  # ✅ KEEP — API_VERSION, SUPABASE_BUCKETS, ANTHROPIC_MODELS
├── enums/
│   └── index.ts                # 🆕 ADD — AppointmentStatus, UserRole, etc.
└── utils/
    └── index.ts                # 🆕 ADD — ONLY cross-platform utils (formatDate, formatCurrency). Remove cn() (frontend-only)
```

### What goes where:
- **`types/`** — Pure TypeScript interfaces/types. No runtime code.
- **`schemas/`** — Zod schemas + derived types + error classes. Runtime validation.
- **`constants/`** — `as const` arrays and objects. Both frontend and backend read these.
- **`enums/`** — String literal union types or const enums for status values, roles, etc.
- **`dto/`** — Request/response shapes if you have complex transformation layers (currently not needed).
- **`utils/`** — Only functions used by BOTH sides. No frontend-only or backend-only utils.

## 12. Final Report

### Current Architecture
```
shared/
├── index.ts              ❌ Unused barrel
├── schemas/
│   ├── index.ts          ⚠️ Redundant barrel
│   └── scan.schema.ts    ✅ Required (error classes + types used by backend)
├── types/
│   └── index.ts          ⚠️ 3 of 9 exports used, 6 are dead code
├── constants/
│   └── index.ts          ⚠️ 2 of 8 exports used, 6 are dead code
└── utils/
    └── index.ts          ❌ 0 of 4 exports used (entirely dead)
```

### Problems Found
1. **Frontend never imports from shared** — tsconfig alias exists but is unused
2. **Frontend duplicates constants** — `LOCALITIES`, `SERVICES`, `BOOKING_SERVICES`, `TIME_SLOTS`, `BOOKED_SLOTS` defined locally
3. **Frontend duplicates `cn()` utility** — identical copy in `src/lib/utils.ts`
4. **60% of shared code is dead** — unused types, constants, utilities, and Zod schemas
5. **Zod validation not wired** — schemas exist but `validate` middleware is never used in routes
6. **Root barrel export unused** — backend imports via relative paths, not from `@glamspot/shared`
7. **Backend uses relative paths** (`../../../shared/`) instead of the package alias `@glamspot/shared`

### Unused Files (Safe to Remove)
- `shared/utils/index.ts` — 0 imports
- `shared/index.ts` — 0 imports  
- `shared/schemas/index.ts` — redundant (backend imports scan.schema directly)

### Unused Exports (Safe to Remove from Their Files)
- 6 types: `Salon`, `Stylist`, `Testimonial`, `BookingService`, `BookingStylist`, `TourSalon`, `ExclusiveService`
- 6 constants: `LOCALITIES`, `SERVICES`, `BOOKING_SERVICES`, `TIME_SLOTS`, `BOOKED_SLOTS`, `API_VERSION`
- 4 utils: `cn`, `formatDate`, `formatCurrency`, `generateTimeSlots`
- 8 Zod schemas (keep the inferred types, remove schema objects if validation won't be used)

### Improvements Recommended
1. **Frontend should import constants from shared** instead of duplicating locally
2. **Wire up Zod validation middleware** in backend routes (or remove schemas)
3. **Backend should use `@glamspot/shared` alias** instead of relative paths
4. **Remove dead code** — ~60% of shared is unused
5. **Add `workspaces` field** to root `package.json` for proper npm workspace resolution
6. **Consider adding an `enums/` folder** for `AppointmentStatus`, `UserRole`, etc.

### Final Recommended Structure
```
shared/
├── package.json           # Keep (with workspaces fix)
├── tsconfig.json          # Keep
├── schemas/
│   ├── index.ts           # Keep (re-export)
│   └── scan.schema.ts     # Keep (error classes + scan types — heavily used)
├── types/
│   ├── index.ts           # Keep (slim down to used types)
│   └── models.ts          # Profile, Appointment, ScanResult (keep)
├── constants/
│   ├── index.ts           # Keep (re-export)
│   ├── locations.ts       # LOCALITIES, SERVICES (frontend should migrate here)
│   ├── booking.ts         # BOOKING_SERVICES, TIME_SLOTS, BOOKED_SLOTS
│   └── api.ts             # SUPABASE_BUCKETS, ANTHROPIC_MODELS, API_VERSION
└── utils/
    └── format.ts          # formatDate, formatCurrency (keep if both sides need them)
```

**Delete:** `shared/utils/index.ts` (cn, generateTimeSlots), `shared/index.ts` (root barrel), 7 unused interfaces, 7 unused Zod schema objects.
